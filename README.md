This project aimed to develop a Mobility service that allows students and staff to rent bicycles, electric scooters, and e‑bikes. These rentals are billed based on usage time and may include additional services such as a helmet or a light. It is also important to note that different types of users (students, staff) benefit from distinct pricing rules.

To achieve this, several functionalities had to be implemented, such as loading users and vehicles from a text file, as well as loading rentals from an object file (if it exists). After these data loads, and by requesting certain parameters from the user, it became possible to implement the functionality to create a rental and to list it, along with all previously created rentals. Finally, we implemented the functionality to export all rentals to an object file, which is created/read during the initial loading process.

 - Utilizador Class and Subclasses: The Utilizador class is a superclass that defines the attributes student/staff ID number and current location, both common to all objects of this type in the system. This allows any user to be referenced generically within the UCMobility class. From this superclass, we created the Estudante and Funcionario classes, which inherit its attributes and methods. Additionally, the Funcionario class adds the attribute contract year, common to all staff members. This class is then subdivided into FuncDocente and FuncNaoDocente, which include the attributes faculty (for FuncDocente) and work service (for FuncNaoDocente). This hierarchical organization enables correct use of polymorphism when handling users in the program, allowing a reference of type Utilizador to represent any specific type (Estudante, FuncDocente, or FuncNaoDocente).

 - VeiculoAluger Class and Subclasses: The VeiculoAluger class is a superclass that defines the common attributes of all vehicle objects in the system, namely the identifier and current location. This allows any vehicle to be referenced generically within UCMobility. The Bicicleta and VeiculoEletrico classes are derived from this superclass, inheriting its attributes and methods. For electric vehicles, the VeiculoEletrico class adds the battery level attribute, common to all vehicles of this type. This class is then subdivided into Trotinete and Ebike, which include the attributes withLCD (for Trotinete) and fixedFrame (for Ebikes). This hierarchical structure enables correct use of polymorphism when handling vehicles in the program, allowing a reference of type VeiculoAluguer to represent any specific type (Bicicleta, Trotinete, or Ebike).

 - Aluguer Class: Using the Aluguer class, we create an object of this type in the main program, representing the rental of a vehicle by a user. To perform this rental, the class defines essential attributes such as the user, the vehicle, the start and end times of the rental, and any additional services requested by the user. These attributes are crucial for executing one of the most important parts of the project: calculating the rental cost.

In this method, we begin by checking the user type, since a 50% discount may apply. After this verification, we calculate the total number of rental hours and compute the cost based solely on usage time, depending on the price of the vehicle type involved. Then, we add the cost of any extra services requested by the user. Finally, as mentioned earlier, we check the user type again to determine whether they are eligible for a discount.

 - UCMobility Class and Ficheiro Class: At the core of the project is the UCMobility class. When the program starts, three methods from the Ficheiro class are called: lerFicheiroUtilizador, lerFicheiroVeiculos, and lerFicheiroAluguer. The lerFicheiroUtilizador method correctly reads the text file containing user information and returns a list with that data. Similarly, lerFicheiroVeiculos reads the text file containing vehicle information and returns a list with that data. The lerFicheiroAluguer method ensures the correct reading of the object file, returning a list of all previously created rentals.

Whenever a new rental is created, the guardaAlugueres method updates the object file containing the list of all rentals, ensuring that all are saved and available for future executions.

After reading the files, the program asks the user for their information: ID number, desired vehicle ID, payment method, and intended rental duration. Once the rental duration is received, the program checks the vehicle’s availability for that time period, since the same vehicle cannot be rented simultaneously by two users. Only after this verification do we retrieve, from the initial arrays, the vehicle and user data needed to perform the rental.

Next, the program asks the user whether they want additional services. These services only differ in the case of a two‑person bicycle, which may require two helmets—something not applicable to the other vehicles.

Using the vehicle and user data, we then create the corresponding objects of the appropriate types defined in the files. With this, we can proceed with the creation of the rental, since all necessary data is now available. Finally, we insert the newly created rental into the initial rental array, display the total cost of all rentals, and conclude the process by exporting the rentals to the rental file using the previously mentioned function.
